# 寻味中国 - 项目总规划书

> 爱吃才会赢 · 中国美食地图
>
> 赛道：生活娱乐 | 项目类型：Android App + Web 后端 | 版本：v1.0

---

## 一、项目概述

「寻味中国」是一款以中国地图为核心交互的美食探索应用，面向厨房新手、美食爱好者和异乡游子，提供省份美食浏览、AI 智能推荐（天气/食材）、制作教程、吃遍中国点亮地图等核心功能。产品同时提供 **Android 原生 App** 和 **Web 端**（已通过报名审核）。

### 1.1 产品架构

产品采用前后端分离架构，后端提供 RESTful API + SSE 流式推送（AI 对话），前端分两套客户端共享同一套后端服务：

| 客户端 | 说明 |
|---|---|
| Android App | 主力客户端，底部三栏导航：对话 / 首页 / 我的，面向日常使用场景 |
| Web 网页版 | 已完成报名 Demo 版，后续迭代为完整功能版，用于演示和轻量访问 |
| 后端服务 | Spring Boot 单体应用，提供用户、美食、AI 对话、排行榜等全部 API |

## 二、技术栈选型

技术选型综合考虑已有技术背景（Java/Spring Boot、Python/LangChain、MySQL/Redis）以及比赛开发效率。

### 2.1 总体技术栈

| 层级 | 技术 | 选型理由 |
|---|---|---|
| **移动端** | Android (Kotlin + Jetpack Compose) | Google 官方推荐的现代 Android 开发方案，声明式 UI 开发效率高 |
| **Web 端** | Vue 3 + Vite + ECharts | 报名版已用 HTML + ECharts 地图，Vue 3 可快速包装为 SPA |
| **后端框架** | Spring Boot 3 + Spring AI | 核心栈，Spring AI 直接对接大模型，支持 Tool Calling 和 SSE 流式输出 |
| **AI 能力** | Spring AI + LangChain4j | Agent 编排、Tool 调用、RAG 检索 |
| **数据库** | MySQL 8 + Redis | MySQL 存业务数据，Redis 缓存热点数据 |
| **文件存储** | MinIO / 阿里云 OSS | 开发阶段用 MinIO 本地部署 |
| **地图服务** | 高德地图 SDK / API | Android 端用高德 SDK，Web 端用 ECharts 地图组件 |
| **大模型** | 通义千问 / DeepSeek / 智谱 GLM | 国内模型，Spring AI 已有适配，支持 Function Calling |
| **构建工具** | Maven (后端) + Gradle (Android) | 标准工具链 |

### 2.2 Android 端技术细节

| 类别 | 技术选型 | 用途 |
|---|---|---|
| UI 框架 | Jetpack Compose + Material 3 | 声明式 UI |
| 网络请求 | Retrofit2 + OkHttp | REST API 调用 |
| 图片加载 | Coil | 美食图片/头像加载与缓存 |
| 地图 | 高德地图 Android SDK | 中国地图渲染与省份交互 |
| AI 对话 | OkHttp SSE + WebSocket | 流式接收 AI 回复 |
| 本地存储 | DataStore / Room | 用户偏好、本地缓存 |
| 依赖注入 | Hilt | 解耦模块 |
| 异步 | Kotlin Coroutines + Flow | 异步网络请求与数据流 |

### 2.3 后端技术细节

| 类别 | 技术选型 | 用途 |
|---|---|---|
| Web 框架 | Spring Boot 3.2 + Spring MVC | RESTful API |
| ORM | MyBatis-Plus | 数据库操作，代码生成器快速建 CRUD |
| AI 集成 | Spring AI + LangChain4j | 大模型调用、Tool Calling、RAG |
| 认证 | JWT + Spring Security | 用户登录 Token 鉴权 |
| 缓存 | Redis + Spring Cache | 排行榜、美食详情、AI 对话上下文 |
| 文件上传 | MinIO SDK | 图片上传管理 |
| API 文档 | Knife4j (Swagger) | 在线 API 文档 |
| 数据库迁移 | Flyway | 版本化数据库表结构管理 |

> **选型原则：** 全部优先选择已有实战经验的技术栈，最大化开发效率。不引入学习成本高的新框架，把精力集中在业务逻辑和用户体验打磨上。

## 六、AI 模块设计

### 6.1 AI 功能矩阵

| 功能 | 输入 | AI 处理 | 输出 |
|---|---|---|---|
| **天气推荐** | 用户位置 → 调天气 API → 温度/天气 | 根据天气条件匹配美食标签，调用大模型生成推荐理由 | 3-5 道推荐菜品 + 推荐语 |
| **冰箱剩菜** | 用户输入食材列表 | 从 food 表检索含对应食材的菜谱 + 大模型补充搭配建议 | 匹配菜品列表 + 推荐新菜式 |
| **营养分析** | 菜品照片（多模态识别） | 调用多模态模型识别菜品 → 匹配营养成分数据 | 蛋白质/脂肪/热量估算 |
| **自由问答** | 用户自由对话 | 多轮对话 + Tool Calling（查菜谱/查营养） | 流式文本回复 |

### 6.2 AI Agent 架构

```
用户输入 → 意图路由 Router
              ├── 天气 → 天气API获取温湿度 → 大模型
              ├── 食材 → 食材匹配MySQL检索 → 大模型
              ├── 营养 → 多模态识别菜品识别 → 大模型
              └── 闲聊 → 大模型直接回复
                                  ↓
                          Tool Calling（查菜谱/查营养）
                                  ↓
                          SSE 流式输出 → 用户
```

## 七、开发计划

### Phase 1：后端基础 + 数据初始化（预计 3-4 天）

- Spring Boot 项目脚手架搭建
- MySQL 建表 + Flyway 迁移脚本
- Redis 配置 + 缓存工具类
- Python 脚本调用大模型批量生成 34 省份美食数据并导入数据库
- MyBatis-Plus 代码生成 + 基础 CRUD API
- JWT 认证 + 用户注册登录

> **里程碑：** 后端服务可启动，API 可调试

### Phase 2：美食核心功能 + Android 首页（预计 4-5 天）

- Android 项目脚手架（Kotlin + Compose + Hilt）
- 底部导航栏（对话 / 首页 / 我的）
- 首页：高德地图 SDK 接入，省份点击交互
- 省份美食列表页 + 美食详情页
- 搜索功能 + 收藏功能
- 探索进度：浏览后省份点亮

> **里程碑：** Android 端地图浏览 + 美食查看完整可用

### Phase 3：AI 对话功能（预计 3-4 天）

- 后端：Spring AI 对接大模型 API
- AI 对话 API（SSE 流式输出）
- 意图路由：天气推荐 / 食材匹配 / 自由问答
- Tool Calling：查菜谱、查食材数据库
- Android 对话页面：聊天界面 + 流式渲染 + 历史记录
- 对话上下文管理（Redis 缓存）

> **里程碑：** AI 对话核心功能完整可用

### Phase 4：排行榜 + 个人中心 + Web 迭代（预计 3-4 天）

- 排行榜功能（Redis ZSET + 定时更新）
- 个人中心页面（头像、昵称、点亮进度地图）
- 评论/点赞功能
- Web 端升级为 Vue 3 完整版（对接后端 API）
- 分享功能（生成海报/链接）

> **里程碑：** 全部功能开发完毕

### Phase 5：打磨优化 + 提交（预计 2-3 天）

- UI/UX 细节打磨（动画、过渡、空状态）
- 性能优化（图片懒加载、列表分页、接口响应速度）
- Bug 修复 + 边界处理
- 录制演示视频 + 整理提交材料
- 部署 Web 端到云服务器（可选）

> **里程碑：** 比赛提交

> **总周期估算：** 约 15-20 天。建议 Phase 1-3 优先保障，Phase 4 的 Web 端可根据时间灵活调整。

## 九、风险与应对

| 风险 | 等级 | 应对策略 |
|---|---|---|
| 美食数据量大，手动录入耗时 | 中 | 用 Python + 大模型 API 批量生成结构化数据，每个省 3-5 道菜即可 |
