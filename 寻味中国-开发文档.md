## 一、可行性分析

### 1.1 技术可行性

| 维度 | 评估 | 说明 |
|---|---|---|
| **后端框架** | ✅ 高 | Spring Boot 3 + MyBatis-Plus 团队已有实战经验，文档与社区成熟 |
| **AI 框架** | ⚠️ 中 | Spring AI 仍处于快速迭代期（1.0 前），需锁定稳定版本；LangChain4j 兜底处理 Tool Calling 复杂场景 |
| **Android UI** | ⚠️ 中 | Jetpack Compose 声明式 UI 上手成本可控，但团队经验"不足"为规划书自评，需 1-2 天专项预热 |
| **地图集成** | ⚠️ 中 | 高德 SDK 官方仅有 View 体系，无原生 Compose 组件，需 `AndroidView` 包裹，学习/调试成本存在 |
| **SSE 流式** | ✅ 高 | Spring MVC 的 `SseEmitter` 成熟，OkHttp SSE 客户端实现简单 |
| **Web 端** | ✅ 高 | 报名版已基于 ECharts 完成，Vue 3 + Vite 升级是常规工作量 |

**技术结论**：技术栈选型合理，无阻塞性技术风险，AI 与地图为两个需重点投入的方向。

### 1.2 时间可行性

| 阶段 | 规划工时 | 风险点 | 调整建议 |
|---|---|---|---|
| Phase 1 后端基础 | 3-4 天 | 34 省 × 3-5 道菜的 AI 数据生成与清洗质量 | 优先保证 34 道"招牌菜"数据落地，扩展菜下个迭代补 |
| Phase 2 美食核心 | 4-5 天 | Compose + 高德地图组合首次接触 | 提前 0.5 天做地图 Demo 验证 |
| Phase 3 AI 对话 | 3-4 天 | Spring AI 版本变更、Tool Calling 调通 | 准备 LangChain4j 备选实现 |
| Phase 4 排行榜/中心/Web | 3-4 天 | 三个子模块并行 | 评审分享、评论功能可推迟到 v1.1 |
| Phase 5 打磨提交 | 2-3 天 | UI 细节无止境 | 提前 1 天冻结功能，仅修 Bug |

**时间结论**：15-20 天**紧但可行**。建议将 Web 端 Vue 完整版**降级为最小可用**，重心放在 Android 端（主力客户端）。

### 1.3 资源可行性

| 资源 | 评估 | 备注 |
|---|---|---|
| 大模型 API | ⚠️ 中 | 比赛 demo 流量小，使用各家免费额度可覆盖；上线后需考虑预算 |
| 高德地图 | ✅ 高 | 个人开发者免费配额足够 |
| 对象存储 | ✅ 高 | MinIO 本地免费；OSS 学习用免费额度 |
| 服务器 | ⚠️ 中 | 比赛若需公网演示，至少 1 台 2C4G 云服务器（约 ¥50-100/月） |
| 人力 | ⚠️ 中 | 假设 1-2 人全职/兼职，时间紧任务重 |

**资源结论**：开发期零成本，比赛演示期云服务器 ¥100 以内可控。

### 1.4 范围可行性

| 子系统 | 范围 | MVP 建议 |
|---|---|---|
| 美食浏览 | 34 省 + 100+ 菜品 | ✅ 必做 |
| AI 对话（4 类） | 天气/食材/营养/自由 | ⚠️ MVP 阶段砍掉"营养分析"多模态 |
| 探索点亮 | 34 省解锁 | ✅ 必做（核心玩法） |
| 排行榜 | 收藏榜 / 浏览榜 | ⚠️ MVP 仅做收藏榜 |
| 评论点赞 | UGC | ❌ 推到 v1.1 |
| Web 端完整版 | 全功能 | ⚠️ 仅保证首页 + 地图 Demo 可用 |

**范围结论**：建议将 v1.0 范围收敛为"**Android 端完整 + Web 端 Demo**"，确保比赛演示质量。

### 1.5 综合结论

✅ **项目整体可行**，但需要在以下三处做"减法"以保证交付质量：

1. **AI 砍掉多模态营养分析**，先做"天气推荐 + 食材匹配 + 自由问答"三类对话；
2. **Web 端保持 Demo 级**，不追求 Vue 3 完整版；
3. **UGC（评论/点赞）推到 v1.1**，避免分散精力。

> 后续章节按收敛后的范围编写。

### 3.5 关键数据流

**美食浏览流：**
```
User → Android 首页 → GET /api/provinces → 高亮已探索省份
  → 点击省份 → GET /api/provinces/{id}/foods
  → 点击美食 → GET /api/foods/{id} + POST /api/foods/{id}/view
  → 后端更新 explore_progress + 收藏计数 → Redis ZSET 累加
```

**AI 对话流：**
```
User 输入 → POST /api/chat/send
  → IntentRouter 分类
  → 调对应 Tool（天气API/MySQL/VectorStore）
  → 拼装 Prompt → LLM
  → SSE 推流 → 前端流式渲染
  → 完整消息入库 + 上下文写 Redis
```

## 四、数据库设计详解

### 4.1 建表脚本约定

- 引擎：`InnoDB`，字符集：`utf8mb4`，排序规则：`utf8mb4_unicode_ci`
- 主键：`BIGINT AUTO_INCREMENT`
- 时间字段：`created_at`、`updated_at` 统一维护
- 软删字段：MVP 阶段不引入，按需添加
- 表注释：每张表必须有 COMMENT

### 4.2 索引策略

| 表 | 索引建议 |
|---|---|
| user | `uk_username`、`uk_phone` |
| food | `idx_province_id`、`idx_status`、`idx_collect_count`（排行榜用） |
| user_favorite | `uk_user_food(user_id, food_id)`、`idx_user_id` |
| explore_progress | `uk_user_province(user_id, province_id)`、`idx_user_id` |
| chat_session | `idx_user_id`、`idx_updated_at` |
| chat_message | `idx_session_id`、`idx_created_at` |

### 4.3 数据初始化策略

**省份数据**：手工整理 34 个省级行政区，写入 `V2__seed_province.sql`。

**美食数据**：使用 Python 脚本 `scripts/generate_food_data.py` 调大模型生成：

```python
# 伪代码
provinces = load_provinces()  # 34 个省
for p in provinces:
    foods = llm.generate(f"请为{p.name}推荐5道最具代表性的菜，JSON 格式返回...")
    # 校验字段、补默认值
    insert_into_db(foods)
```

要求：
- 每省 3-5 道招牌菜（合计 100-170 条）
- 字段完整性校验（name/ingredients/steps 必填）
- 失败重试 + 人工兜底
- 一次性生成 V2/V3 种子脚本入库

### 4.4 Flyway 迁移文件命名

```
db/migration/
├── V1__init_schema.sql       # 7 张表
├── V2__seed_province.sql     # 34 省数据
├── V3__seed_food.sql         # 美食数据
└── V4__add_index.sql         # 后续优化索引
```

## 五、API 设计规范

### 5.1 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": { ... },
  "timestamp": 1715000000000
}
```

| 字段 | 含义 |
|---|---|
| code | 0 成功，非 0 失败 |
| message | 提示信息 |
| data | 业务数据，可为 null |
| timestamp | 服务端时间戳 |

### 5.2 错误码规范

| 段 | 范围 | 含义 |
|---|---|---|
| 0 | 0 | 成功 |
| 1xxx | 1000-1999 | 通用错误（参数错误 1001、未授权 1002、禁止访问 1003、资源不存在 1004） |
| 2xxx | 2000-2999 | 用户模块（用户名已存在 2001、密码错误 2002） |
| 3xxx | 3000-3999 | 美食模块（菜品已下架 3001） |
| 4xxx | 4000-4999 | AI 模块（模型调用失败 4001、上下文超限 4002） |
| 5xxx | 5000-5999 | 系统错误（数据库 5001、Redis 5002、文件存储 5003） |

### 5.3 分页规范

请求：
```
GET /api/foods?pageNum=1&pageSize=10&keyword=火锅
```

响应：
```json
{
  "code": 0,
  "data": {
    "list": [...],
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "pages": 10

}
```

### 5.5 SSE 流式接口规范

请求：
```
POST /api/chat/sessions/123/send
Content-Type: application/json
Authorization: Bearer xxx

{"content": "今天吃什么"}
```

响应（`text/event-stream`）：
```
event: message
id: 1
data: {"delta":"今天","done":false}

event: message
id: 2
data: {"delta":"适合","done":false}

event: message
id: 3
data: {"delta":"吃","done":false}

event: finish
data: {"sessionId":123,"messageId":456,"done":true}
```

事件类型：
- `message`：增量文本片段
- `tool`：工具调用过程（可选）
- `finish`：会话结束
- `error`：异常

## 六、AI 模块详细设计

### 6.1 意图路由

```java
public enum IntentType {
    WEATHER,    // 天气推荐
    INGREDIENT, // 食材匹配
    NUTRITION,  // 营养分析（v1.1）
    CHAT        // 自由对话
}
```

**实现思路**：
1. 调用小模型（如 qwen-turbo）做意图分类，输出枚举
2. Prompt 模板：

```
你是意图分类器，根据用户输入判断属于以下哪类：
- WEATHER：询问根据天气/温度/季节推荐菜品
- INGREDIENT：用户提供具体食材询问做法
- NUTRITION：询问菜品营养/热量
- CHAT：其他闲聊或综合问题

只返回枚举值，不要解释。

用户输入：{userInput}
```

### 6.2 天气推荐实现

```
1. 调高德天气 API（用户授权位置）→ 获取 temperature/weather
2. 拼装 Prompt：
   "当前城市：{city}，温度：{t}℃，天气：{w}。请从以下菜品中推荐 3-5 道，
    并给出推荐理由。菜品列表：{foodList}"
3. 大模型输出 → SSE 流式推送
4. 解析推荐结果 → 关联 food.id → metadata 字段存储
```

### 6.3 食材匹配实现

```
1. 提取用户输入中的食材（简单关键词匹配 + 实体识别）
2. SQL 检索：SELECT * FROM food WHERE ingredients LIKE '%番茄%' OR ...
3. 取 Top 10 菜品作为上下文
4. 大模型基于候选菜品生成推荐理由
5. 输出匹配菜列表 + 步骤简述
```

### 6.4 多轮对话管理

**Redis Key 设计**：
```
chat:context:{sessionId}  →  List<ChatMessage>，保留最近 10 轮
```

**策略**：
- 用户发送消息 → 写入 Redis List 头
- 超过 10 条 → 弹出队尾
- 拼装 messages 数组 → 调用 LLM
- 完整回复入库 → 同步写 Redis

### 6.5 Prompt 模板库

文件位置：`resources/prompts/`

```
system-base.txt          # 基础人设
weather-recommend.txt    # 天气推荐
ingredient-match.txt     # 食材匹配
free-chat.txt            # 自由对话
intent-classify.txt      # 意图分类
```

| 工具名 | 用途 | 入参 | 出参 |

### 7.5 SSE 客户端实现

```kotlin
class SseClient(private val okHttpClient: OkHttpClient) {
    suspend fun stream(
        url: String,
        body: RequestBody,
        onEvent: (SseEvent) -> Unit
    ) = withContext(Dispatchers.IO) {
        val request = Request.Builder()
            .url(url)
            .post(body)
            .header("Accept", "text/event-stream")
            .build()
        okHttpClient.newCall(request).execute().use { response ->
            val source = response.body?.source() ?: return@use
            // 解析 event:/data:/id:/空行 格式
            while (!source.exhausted()) {
                val line = source.readUtf8Line() ?: break
                // 解析逻辑
            }
        }
    }
}
```
